# ARGOS Skill Creation Protocol

## REGULA ZERO

Inainte sa creezi skill nou, intreaba: exista deja unul similar?
- Query: `SELECT id, path, name FROM skills_tree WHERE path ILIKE '%keyword%' OR name ILIKE '%keyword%';`
- Daca exista skill similar care acopera 60%+ din ce vrei sa documentezi -> UPDATE existing, NU creezi duplicate
- Daca exista skill cu path identic -> UPDATE obligatoriu (path are UNIQUE constraint, INSERT va pica)
- Skill nou creat doar cand subiectul e clar distinct si nu se suprapune cu nimic existent

## NAMING CONVENTION

**Path format:** `parent-category/skill-specific-name`
- Lowercase strict
- Cuvinte separate prin hyphen (`-`), NU underscore, NU space
- Parent category descriptiv si scurt: `argos-core/`, `argos-deploy/`, `docker-swarm/`, `nixos/`, `postgresql/`, `file-ops/`, `argos-agent/`, `argos-meta/`
- Skill name explicit despre ce face: `argos-api-redeploy`, NU `redeploy` sau `api-stuff`
- Path UNIQUE in DB - verifica inainte de INSERT

**Name field (separate from path):** descriptiv human-readable, max 200 chars. Poate avea spatii si majuscule.
- Exemplu: path=`argos-deploy/argos-api-redeploy`, name=`ARGOS API Redeploy Protocol`

## CONTENT STRUCTURE

Standard markdown sections (in this order):

1. **Title H1** - skill name as readable title
2. **REGULA ZERO** (optional, for emergency/critical skills) - the one rule that matters most
3. **SCOP** sau **PURPOSE** - 2-3 sentences, what this skill solves
4. **PATTERN** sau **PROTOCOL** - the actual procedure, code examples, commands
5. **EXEMPLE** - concrete usage cases when pattern is non-obvious
6. **ANTI-PATTERNS** - what NOT to do, with explicit "NU foloseste X" lines
7. **VERIFICARE** - how to confirm the procedure worked
8. **VEZI SI** - cross-references to related skills with id and path

Length guideline: 1500-8000 chars for normal skills. Emergency skills can go to 20K (skill #90 self-knowledge is 21K). Avoid skills under 500 chars - they don't justify the slot.

## TAGS CONVENTIONS

Tags array is critical for agent loop filtering and keyword scoring.

**Phase tags** (for agent loop filtering):
- `phase:planning` - skill used during agent planning phase (architecture, design decisions)
- `phase:executing` - skill used during command execution (deploy, ops, file operations)
- `phase:verifying` - skill used during result verification (health checks, state validation)
- `phase:fixing` - skill used during error recovery (debug, restore, rollback)
- A skill can have multiple phase tags if cross-cutting (skill #90 has all 4)

**Semantic tags:**
- Domain keywords: `docker`, `swarm`, `nixos`, `postgresql`, `ssh`, `api`, etc
- Action keywords: `deploy`, `redeploy`, `restart`, `backup`, `restore`
- Quality flags: `critical`, `gotcha`, `reference`

**Special tags:**
- `emergency` - skill must be loaded in agent system prompt regardless of phase (use sparingly, see emergency flag below)
- `auto-generated` - skill was created by Claude/automation, not handwritten by darkangel
- `meta` - skill about skills/tools/processes (this skill has it)

## SOURCE FIELD

`skills_tree.source` column has CHECK constraint: `source IN ('manual', 'auto')`.
- `source='manual'` - written by darkangel manually
- `source='auto'` - generated by Claude or other automation
- ALWAYS set source explicitly on INSERT, do not rely on default

## INSERT PATTERN (skill #91 reference)

The ONLY correct way to insert a skill with multiline content is Python + asyncpg + triple-quoted CONTENT + $1 binding. NEVER heredoc SQL via `docker exec psql` (escape hell, see skill #72).

Pattern:

```python
python3.13 << PYEOF_MARKER
import asyncio, asyncpg, os
from dotenv import load_dotenv
load_dotenv(os.path.expanduser("~/.argos/argos-core/config/.env"))

CONTENT = """# Skill Title Here

Skill content with apostrophes 'like this' work fine.
SQL examples with backticks work fine.
Multiple paragraphs OK.
"""

async def main():
    pool = await asyncpg.create_pool(
        host=os.getenv("DB_HOST", "11.11.11.111"),
        port=int(os.getenv("DB_PORT", "5432")),
        user=os.getenv("DB_USER"),
        password=os.getenv("DB_PASSWORD"),
        database=os.getenv("DB_NAME"),
    )
    async with pool.acquire() as conn:
        # Pre-check: does path already exist?
        existing = await conn.fetchrow(
            "SELECT id FROM skills_tree WHERE path = $1",
            "parent/skill-name"
        )
        if existing:
            print(f"[ABORT] path already exists with id={existing['id']}")
            await pool.close()
            return
        # Insert new skill
        new_id = await conn.fetchval(
            """
            INSERT INTO skills_tree
                (path, parent_path, name, content, tags, source,
                 emergency, verified, verified_by, verified_at, created_at)
            VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9, NOW(), NOW())
            RETURNING id
            """,
            "parent/skill-name",
            "parent/",
            "Skill Display Name",
            CONTENT,
            ["tag1", "tag2", "phase:executing"],
            "auto",
            False,  # emergency
            True,   # verified
            "claude",  # verified_by
        )
        # Verify
        row = await conn.fetchrow(
            "SELECT id, path, LENGTH(content) AS sz, array_length(tags, 1) AS tag_count FROM skills_tree WHERE id = $1",
            new_id
        )
        print(f"[OK] inserted skill #{row['id']}: {row['path']} ({row['sz']} chars, {row['tag_count']} tags)")
    await pool.close()

asyncio.run(main())
PYEOF_MARKER
```

**GOTCHA on triple-quote nesting:** if CONTENT contains triple-double-quote literal (e.g. Python code examples), use triple-SINGLE-quote as outer (`'''...'''`). If CONTENT contains triple-single, use triple-DOUBLE outer. Never mix both inside same CONTENT.

## VERIFIED FLAG

- `verified=TRUE` - skill content tested live, commands actually executed and confirmed working
- `verified=FALSE` - draft skill, content not yet verified against reality
- `verified_by` - who verified ('darkangel', 'claude', 'auto')
- `verified_at` - timestamp of verification

Default for newly created skills: depends on context. If you're documenting an existing working pattern, verified=TRUE. If you're drafting a procedure that hasn't been tested yet, verified=FALSE and ask darkangel to verify before use.

## EMERGENCY FLAG

Use SPARINGLY. As of April 2026, only 5 skills are emergency: #88 (api-redeploy), #89 (nixos-systemd-path-gotcha), #90 (self-knowledge), #91 (output-patterns), #92 (agent-loop-architecture).

Emergency means: this skill is automatically loaded in the agent's system prompt regardless of keyword matching or phase filtering. It's always present.

**Criteria for emergency=TRUE:**
- Skill is critical for safety or correctness in many scenarios
- Not knowing this skill leads to data loss, broken deploys, or wasted hours
- Skill is referenced by other skills frequently

**Default: emergency=FALSE.** If unsure, do not set emergency. It can be promoted later.

## ANTI-PATTERNS

- NU folosi heredoc SQL via `docker exec psql` for INSERT - escape hell, see skill #72
- NU folosi `sed` for editing skill content already in DB - use Python read/modify/write or asyncpg UPDATE with $1 binding
- NU duplicate paths - they have UNIQUE constraint, INSERT will fail
- NU create skills with content under 500 chars unless they document one specific gotcha worth its own slot
- NU set emergency=TRUE casually - 5 skills max in normal operation, promotion is intentional
- NU forget source='auto' on Claude-generated skills - audit trail matters
- NU mix phases - if skill is truly cross-phase, add multiple phase tags. If unclear, omit phase tags entirely
- NU write skills in romana for new content - English only for code/db/skill content (Session 2 rule)
- NU paste content with diacritics or smart quotes (curly quotes from word processors) - they break grep/regex matching later
- NU insert without pre-check on path uniqueness - wastes time on rollback

## VERIFICATION AFTER INSERT

After INSERT, always run:

```sql
SELECT id, path, name, LENGTH(content) AS content_size,
       array_length(tags, 1) AS tag_count,
       emergency, verified, source, parent_path, created_at
FROM skills_tree
WHERE id = <new_id>;
```

Confirm:
- content_size matches expected (within ~50 chars of what you wrote)
- tag_count matches what you set
- parent_path correctly set (if you forgot it on INSERT, UPDATE it separately)
- source is what you intended ('auto' or 'manual')

If you want to verify the skill is queryable for the agent loop:

```sql
-- Test phase filter (should return your skill if you set phase tag)
SELECT id, path FROM skills_tree WHERE 'phase:executing' = ANY(tags) AND id = <new_id>;

-- Test emergency filter (should return your skill if emergency=TRUE)
SELECT id, path FROM skills_tree WHERE emergency=TRUE AND id = <new_id>;
```
